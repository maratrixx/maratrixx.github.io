hystrix-go 之统计控制器实现

2021年1月27日
| Go源码阅读
| 阅读

上一篇 我们分析断路器时有用到 metricExchange，但并未展开详细介绍，本篇我们主要介绍它以及实现原理。

metricExchange 主要是用来收集处理上报的所有事件，并对事件进行汇总处理，最后根据计算出来的一段时间内的错误率来表示服务的健康度。下面我们看下其源码实现。

源码分析

metricExchange

 1type metricExchange struct {
 2	Name    string
 3	Updates chan *commandExecution  // 上报的信息
 4	Mutex   *sync.RWMutex
 5
 6	metricCollectors []metricCollector.MetricCollector  // 统计收集器列表
 7}
 8
 9type commandExecution struct {
10	Types            []string      `json:"types"`             // 事件列表
11	Start            time.Time     `json:"start_time"`        // 开始时间
12	RunDuration      time.Duration `json:"run_duration"`      // 运行耗时
13	ConcurrencyInUse float64       `json:"concurrency_inuse"` // 当前并发量占比
14}

这里我们主要看下 Updates 和 metricCollectors 字段：

  • Updates 是用来接收 *commandExecution 对象信息的，包括开始时间、运行耗时、并发占比
  • metricCollectors 可以包含多个统计收集器，对上报来的运行指标就行收集处理，默认只包含 newDefaultMetricCollector 收集器

Monior 方法

在执行 newMetricExchange 的时候会启动一个协程 go m.Monitor() 去监控 Updates 的数据，然后上报给 metricCollectors 保存执行的信息数据比如前面提到的调用次数，失败次数，被拒绝次数等等。

 1func (m *metricExchange) Monitor() {
 2	for update := range m.Updates {
 3		// we only grab a read lock to make sure Reset() isn't changing the numbers.
 4		m.Mutex.RLock()
 5
 6		totalDuration := time.Since(update.Start)
 7		wg := &sync.WaitGroup{}
 8		for _, collector := range m.metricCollectors {
 9			wg.Add(1)
10			go m.IncrementMetrics(wg, collector, update, totalDuration)
11		}
12		wg.Wait()
13
14		m.Mutex.RUnlock()
15	}
16}

通过上一篇的分析，我们知道在断路器中的 ReportEvent 方法中会进行指标上报：

1circuit.metrics.Updates <- &commandExecution{ // 发送命令执行的指标数据
2    Types:            eventTypes,       // 事件列表
3    Start:            start,            // 开始事件
4    RunDuration:      runDuration,      // 执行耗时
5    ConcurrencyInUse: concurrencyInUse, // 当前的并发占比
6}

而对 circuit.metrics.Updates 的接收便是在 Monitor 方法中，这下便能对应起来了。

Montor 的执行也很简单，首先是遍历 metricCollectors 收集器列表，然后对每个收集器分别执行 go m.IncrementMetrics(wg, collector, update, totalDuration) 进行收集处理。

IncrementMetrics 方法

 1func (m *metricExchange) IncrementMetrics(wg *sync.WaitGroup, collector metricCollector.MetricCollector, update *commandExecution, totalDuration time.Duration) {
 2	// granular metrics
 3	r := metricCollector.MetricResult{
 4		Attempts:         1,
 5		TotalDuration:    totalDuration,
 6		RunDuration:      update.RunDuration,
 7		ConcurrencyInUse: update.ConcurrencyInUse,
 8	}
 9
10	switch update.Types[0] {
11	case "success":
12		r.Successes = 1
13	case "failure":
14		r.Failures = 1
15		r.Errors = 1
16	case "rejected":
17    // ...
18
19	collector.Update(r)
20
21	wg.Done()
22}

IncrementMetrics 对不同的事件进行计数，最后调用收集器的 collector.Update(r) 方法进行汇总，关于默认收集器的实现，我们在下面分析。

IsHealthy 方法

通过之前的分析我们知道，断路器是调用 IsHealthy 方法来判断健康度，然后进行熔断的，我们看下其实现：

 1// 计算最近 10s 内的请求错误率
 2func (m *metricExchange) ErrorPercent(now time.Time) int {
 3	m.Mutex.RLock()
 4	defer m.Mutex.RUnlock()
 5
 6	var errPct float64
 7	reqs := m.requestsLocked().Sum(now)            // 计算过去 10s 内累计的请求数
 8	errs := m.DefaultCollector().Errors().Sum(now) // 计算过去 10s 内累计的请求失败数
 9
10	if reqs > 0 {
11		errPct = (float64(errs) / float64(reqs)) * 100
12	}
13
14	// 向上取整，好奇为什么不直接用 math.Ceil(errPct)
15	return int(errPct + 0.5)
16}
17
18// 根据错误率计算当前健康度，当错误率到达阈值返回 false，未达到阈值返回 true
19func (m *metricExchange) IsHealthy(now time.Time) bool {
20	return m.ErrorPercent(now) < getSettings(m.Name).ErrorPercentThreshold
21}

首先，调用 m.ErrorPercent(now) 获取默认收集器过去 10s 内的请求总数和请求错误数，然后计算出错误率，并做向上取整。

然后，获取 hystrix-go 配置的错误率阈值 ErrorPercentThreshold。

一旦 m.ErrorPercent(now) < getSettings(m.Name).ErrorPercentThreshold 表明当前服务的错误率超过了阈值，即健康度偏低，这样简单粗暴的方式来判断是否需要熔断的。

默认收集器

该仓库除了提供默认的收集器外，还提供了其他几个收集器，其实只要实现了 MetricCollector 接口都可以作为收集器。

1type MetricCollector interface {
2	// Update accepts a set of metrics from a command execution for remote instrumentation
3	Update(MetricResult)
4	// Reset resets the internal counters and timers.
5	Reset()
6}

我们主要看下默认的收集器：

 1type DefaultMetricCollector struct {
 2	mutex *sync.RWMutex
 3
 4	numRequests *rolling.Number // 统计请求总数
 5	errors      *rolling.Number // 统计错误总数
 6
 7	successes               *rolling.Number // 统计请求成功数
 8	failures                *rolling.Number // 统计请求失败数
 9	rejects                 *rolling.Number // 统计请求被拒绝数（触发限流）
10	shortCircuits           *rolling.Number // 统计请求被熔断数
11	timeouts                *rolling.Number // 统计请求超时数
12	contextCanceled         *rolling.Number // 统计请求 ctx 取消数
13	contextDeadlineExceeded *rolling.Number // 统计请求 ctx 超时数
14
15	fallbackSuccesses *rolling.Number // 统计 fallback 成功数
16	fallbackFailures  *rolling.Number // 统计 fallback 失败数
17	totalDuration     *rolling.Timing // 统计所有请求总耗时
18	runDuration       *rolling.Timing // 统计当个请求耗时
19}

这里我们主要看下 *rolling.Number，它才是状态最终保存的地方。

1type Number struct {
2	Buckets map[int64]*numberBucket
3	Mutex   *sync.RWMutex
4}
5
6type numberBucket struct {
7	Value float64
8}

Number 保存了 10 秒内的 Buckets 数据信息，每一个 Bucket 的统计时长为 1 秒。字典字段 Buckets map[int64]*numberBucket 中的 Key 保存的是当前时间（秒时间戳）。

 1// 获取当前时间戳对应的 numberBucket，不存在时创建
 2func (r *Number) getCurrentBucket() *numberBucket {
 3	now := time.Now().Unix()
 4	var bucket *numberBucket
 5	var ok bool
 6
 7	if bucket, ok = r.Buckets[now]; !ok {
 8		bucket = &numberBucket{}
 9		r.Buckets[now] = bucket
10	}
11
12	return bucket
13}
14
15// 移除 10s 以前的 numberBucket
16func (r *Number) removeOldBuckets() {
17	now := time.Now().Unix() - 10
18
19	for timestamp := range r.Buckets {
20		// TODO: configurable rolling window
21		if timestamp <= now {
22			delete(r.Buckets, timestamp)
23		}
24	}
25}
26
27// 增加当前时间对应的 numberBucket 值
28func (r *Number) Increment(i float64) {
29	if i == 0 {
30		return
31	}
32
33	r.Mutex.Lock()
34	defer r.Mutex.Unlock()
35
36	b := r.getCurrentBucket() // 获取当前 numberBucket
37	b.Value += i              // numberBucket.Value += i
38	r.removeOldBuckets()      // 移除 10s 以前的 numberBucket
39}

*Number 除了上面的三个方法，还实现了 UpdateMax、Sum、Max、Avg 方法，代码比较简单，这里不再说明。

默认收集器 DefaultMetricCollector 的 Update 方法才是最终更新统计信息的地方。

 1func (d *DefaultMetricCollector) Update(r MetricResult) {
 2	d.mutex.RLock()
 3	defer d.mutex.RUnlock()
 4
 5	d.numRequests.Increment(r.Attempts)
 6	d.errors.Increment(r.Errors)
 7	d.successes.Increment(r.Successes)
 8	d.failures.Increment(r.Failures)
 9	d.rejects.Increment(r.Rejects)
10	d.shortCircuits.Increment(r.ShortCircuits)
11	d.timeouts.Increment(r.Timeouts)
12	d.fallbackSuccesses.Increment(r.FallbackSuccesses)
13	d.fallbackFailures.Increment(r.FallbackFailures)
14	d.contextCanceled.Increment(r.ContextCanceled)
15	d.contextDeadlineExceeded.Increment(r.ContextDeadlineExceeded)
16
17	d.totalDuration.Add(r.TotalDuration)
18	d.runDuration.Add(r.RunDuration)
19}

可以看到，Update 方法内部其实是调用的 *rolling.Number 的 Increment 方法来更新数据的。
